# Truthwatcher v0.1.0 Quickstart

This quickstart proves the local fixture-backed evidence-to-understanding workflow for a new user. v0.1.0 is local, seeded/fixture-backed, read-only, and not a production discovery collector. It does not contact live devices, perform network writes, handle credentials, call external LLMs, or integrate with external systems.

## Prerequisites

- Go installed.
- PostgreSQL running locally.
- `createdb`, `psql`, `curl`, and `make` available on your PATH.

## 1. Build the binary

```sh
make build
```

Expected result:

```text
bin/truthwatcher
```

## 2. Create the database and configure Truthwatcher

```sh
createdb truthwatcher
export TRUTHWATCHER_DATABASE_URL='postgres://localhost/truthwatcher?sslmode=disable'
```

If the database already exists, `createdb` may report that it exists; reuse it or create a fresh database name and update `TRUTHWATCHER_DATABASE_URL`.

## 3. Run embedded migrations

```sh
./bin/truthwatcher migrate up
```

Expected output includes applied or already-applied embedded migrations, ending with a successful migration status.

## 4. Run fake fixture-backed discovery

```sh
./bin/truthwatcher discover fake --target fixture://junos-mx
```

Expected output resembles:

```text
stored evidence <evidence-id> "show version"
stored evidence <evidence-id> "show chassis hardware"
stored evidence <evidence-id> "show lldp neighbors"
stored evidence <evidence-id> "show bgp summary"
completed discovery run <discovery-run-id> with 4 evidence records
```

Save the `<discovery-run-id>` printed in the last line.

An IOS-XR fixture is also included:

```sh
./bin/truthwatcher discover fake --target fixture://iosxr-asr
```

## 5. Parse the stored discovery run

```sh
./bin/truthwatcher parse discovery-run --id <discovery-run-id> --platform junos
```

Expected output resembles:

```text
parsed discovery run <discovery-run-id>: 4 evidence, 4 assets, 6 facts, 3 relationships, 0 warnings
```

Parsing is intentionally separate from discovery. Raw evidence is stored first and remains available even when parser support is incomplete.

## 6. Start the local server

```sh
./bin/truthwatcher server
```

The server defaults to:

```text
http://127.0.0.1:8080
```

## 7. Open UI routes

Open these routes in a browser:

```text
http://127.0.0.1:8080/
http://127.0.0.1:8080/#/discovery-runs
http://127.0.0.1:8080/#/discovery-runs/<discovery-run-id>
http://127.0.0.1:8080/#/assets
http://127.0.0.1:8080/#/graph
http://127.0.0.1:8080/#/discovery-plans
http://127.0.0.1:8080/#/architecture-seeds
http://127.0.0.1:8080/#/questions
http://127.0.0.1:8080/#/about
```

On `#/discovery-runs/<discovery-run-id>`, inspect the run-level evidence table. It shows the exact command/API, method, target, profile, collection timestamp, raw output hash, and evidence ID for each collected record. Use **Open evidence** to view the read-only raw output drawer; raw output is hidden from the table until you open a specific evidence record.

If the run detail page has evidence but assets or graph views are empty, parse the discovery run first with the CLI command in step 5 or the read-only parser API. The run detail page also links to assets, graph inspection, and audit records filtered for that discovery run.

Seeded architecture context shown in the UI is planning context only. It is not observed proof.

## 8. Basic API checks

In another terminal with `TRUTHWATCHER_DATABASE_URL` still exported, run:

```sh
curl http://127.0.0.1:8080/healthz
curl http://127.0.0.1:8080/api/v1/version
curl http://127.0.0.1:8080/api/v1/discovery-runs
curl http://127.0.0.1:8080/api/v1/discovery-runs/<discovery-run-id>/evidence
curl http://127.0.0.1:8080/api/v1/assets
```

After choosing an asset id from the assets response, check evidence-backed model data:

```sh
curl http://127.0.0.1:8080/api/v1/assets/<asset-id>/facts
curl http://127.0.0.1:8080/api/v1/assets/<asset-id>/relationships
curl http://127.0.0.1:8080/api/v1/assets/<asset-id>/graph
curl 'http://127.0.0.1:8080/api/v1/graph/neighbors?asset_id=<asset-id>'
```

## Audit inspection

After fake discovery, use the read-only audit API or embedded `#/audit` page to inspect who initiated each discovery action, which target/profile/task/command was mapped, and which discovery run or evidence record it relates to. See [Audit records](audit.md).

## 9. One-command local acceptance check

When `TRUTHWATCHER_DATABASE_URL` points at a local PostgreSQL database, run:

```sh
make acceptance-v0.1.0
```

The target runs migrations, executes fake discovery against `fixture://junos-mx`, parses the resulting run, and checks that evidence, assets, and relationship/graph-source data are present. It is boring and local: no Docker, containers, external services, live devices, credentials, or network access are required.

## Safety reminder

For v0.1.0, raw evidence remains the source of truth. Parsed assets, facts, relationships, graph views, and deterministic answers are derived from raw evidence or clearly labeled seeded context.
