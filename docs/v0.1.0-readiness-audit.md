# Truthwatcher v0.1.0 Readiness Audit

Date: 2026-06-25

## Executive Summary

Truthwatcher's current fundamental purpose is to prove an evidence-first, vendor-neutral path from local fixture-backed network observations to explainable network understanding. In the current repository, that means a single Go binary can collect safe fake discovery evidence, persist raw evidence, parse stored evidence into identity candidates, assets, facts, and relationships, expose graph and evidence inspection APIs, and render a small embedded UI for discovery, assets, graph, planning seeds, and deterministic questions.

The product is close to a usable v0.1.0 only if v0.1.0 is explicitly scoped as a local proof-of-concept with PostgreSQL, seeded and fixture-backed data, and no production collector promises. The main release risk is not missing ambition; it is inconsistency between the promised walkthrough/UI/docs and what a new user can actually complete without source-code knowledge.

## 1. Current Fundamental Purpose

Truthwatcher currently exists to make network understanding defensible by preserving raw evidence before deriving identity, assets, facts, relationships, graph views, or answers. Its useful v0.1.0 purpose should be stated as:

> Truthwatcher is a local evidence-first network understanding app that demonstrates how fixture-backed or user-seeded data can become explainable assets, facts, relationships, and graph context while preserving provenance, confidence, uncertainty, and human review boundaries.

Important boundaries:

- It is not yet a production source of truth.
- It is not yet a network automation platform.
- It is not yet a production discovery collector.
- It is not yet an external AI/LLM product.
- It is not yet a cloud, container, or microservice deployment.

## 2. User Workflow That Proves The Purpose Today

The smallest workflow that proves the purpose today is:

1. Build the single binary.
2. Create a local PostgreSQL database and run embedded migrations.
3. Run fake fixture-backed discovery, such as `fixture://junos-mx`.
4. Inspect discovery runs and raw evidence through the API or UI.
5. Parse the stored discovery run with the matching sample parser.
6. Inspect resulting assets, facts, relationships, identity candidates, and graph context.
7. Open the embedded UI to browse discovery runs, assets, evidence drawers, graph neighborhoods, seeded context, safe discovery plans, and deterministic read-only questions.

This proves the purpose because the user can move from local raw fixture output to evidence-linked modeled knowledge without touching a network device.

## 3. Missing Before v0.1.0 Can Be Called Usable

### Release blockers

1. **One-click or one-page demo path is missing.** The repo has fixture data and demo seed SQL, but a new user still has to stitch together build, database setup, migration, discovery, parse, and UI/API inspection manually. v0.1.0 needs a single authoritative quickstart that includes exact commands and expected outputs.
2. **Discovery detail page does not show the evidence list.** The UI route loads evidence for counts but only renders run metadata and seed input. A user must use API endpoints or asset evidence drawers to inspect run-level raw evidence.
3. **No UI identity review page exists.** Identity candidate and review APIs exist, but the embedded UI has no candidate review queue, candidate detail, accept/reject/defer form, or handoff report view.
4. **No UI audit page or audit API exists.** Audit records are persisted for discovery execution, but users cannot inspect them through a first-class endpoint or embedded UI page.
5. **Docs promise at least one endpoint that is not routed.** The POC walkthrough tells users to call `GET /api/v1/relationships`, but the API only exposes asset-scoped relationship and graph routes.
6. **The sample IOS-XR fixture name is inconsistent in CLI help.** Help text mentions `fixture://iosxr-pe`, while the repository fixture and tests use `fixture://iosxr-asr`.
7. **Docs contain stale parser-persistence language.** At least one concept page says parser persistence is future work, while parser persistence now exists.
8. **README and roadmap overstate maturity in places.** They correctly call the repo early-stage, but the v0.1.0 release must avoid implying broad discovery, continuous operation, or enterprise source-of-truth capability.
9. **Acceptance test is not codified as a runnable script.** There is no single `make demo` or `make acceptance-v0.1.0` command that validates the fixture-backed evidence-to-graph loop.
10. **CLI parse implementation has duplicate unreachable error return.** This is minor technically but should be cleaned because it suggests rushed readiness.

### Nice-to-have, not blockers

- More attractive graph visualization.
- Richer fixture corpus.
- More parsers.
- Full-text UI search.
- Auth.
- External import adapters.
- Packaging polish beyond local binary.
- Dedicated documentation site.

## 5. What v0.1.0 Should Explicitly Include

v0.1.0 should include only the smallest coherent local proof:

- Single Go binary.
- Embedded migrations.
- PostgreSQL as the only required database.
- Local CLI for `version`, `server`, `migrate`, `discover fake`, `parse discovery-run`, `devices add/list`, `export json`, `import json`, and `dev check-knowledge`.
- Fake fixture-backed discovery for included Junos and IOS-XR fixtures.
- Raw evidence persistence with metadata and hashes.
- Parser persistence from stored evidence into assets, facts, relationships, parse results, and identity candidates.
- Identity review API with non-destructive decisions and aliases where applicable.
- Discovery audit persistence.
- Asset/fact/relationship/evidence/graph read APIs.
- Embedded UI for dashboard, discovery runs, assets, evidence drawer, graph, plans, seeds, deterministic questions, and about/help.
- Seeded architecture context clearly labeled as user-seeded planning context, not proof.
- Local JSON export and validate-only import.
- A v0.1.0 walkthrough with exact commands, expected output snippets, and limitations.

## 6. What v0.1.0 Should Explicitly Exclude

v0.1.0 should explicitly exclude:

- Network writes, config changes, remediation, commits, reboots, clears, or arbitrary command execution.
- Production SSH/NETCONF/SNMP/gNMI/RESTCONF collector claims.
- Cloud services, containers, microservices, message brokers, Kubernetes, hosted modes, or external services.
- External LLM calls.
- Broad observability, telemetry stacks, tracing systems, or metrics platforms.
- Production authentication and authorization unless already implemented as a small local-only boundary.
- Production integration adapters for IPAM, EMS, Nautobot, NetBox, monitoring, cloud inventory, or source-of-truth systems.
- Continuous discovery scheduling.
- Destructive identity merge or automatic canonical rewrites.
- Raw evidence export by default.

## 7. Docs Needed For A New User

### Required before release

1. **`docs/v0.1.0-quickstart.md` or equivalent README section** with exact database, migration, fake discovery, parse, UI, and API commands.
2. **Expected output appendix** for the fixture-backed POC showing representative discovery run, evidence, asset, fact, relationship, graph, identity candidate, and audit records.
3. **Known limitations page** that states fixture-backed data is acceptable, production collectors are excluded, auth is absent, and seeded context is not proof.
4. **Identity review guide** explaining candidates, strengths, review states, non-destructive accept/reject/defer/more-evidence actions, and aliases.
5. **Audit guide** explaining what is persisted today and how to inspect it after the audit API/UI exists.
6. **UI guide** mapping each route to the evidence-first workflow.
7. **Prompt-pack status page** marking prompts as completed, v0.1.0 cleanup, deferred, rewritten, or archived.

### Useful but not required

- Screenshot-based walkthrough.
- Parser authoring guide.
- Fixture authoring guide.
- Release notes with a copy-paste acceptance test transcript.

## 8. UI Pages Needing Cleanup Before Release

### Release-blocking UI cleanup

- **Discovery Runs detail:** render evidence records, command/API names, hashes, and raw evidence drawer links.
- **Identity Review:** add a page for pending candidates, all candidates, detail, evidence link, action form, and handoff report.
- **Audit:** add a read-only audit inspection page after adding a minimal API route.
- **About:** make it the release help surface with exact v0.1.0 scope, demo commands, safety boundaries, and links to OpenAPI/docs.
- **Graph:** make empty states explain whether the user needs to parse a discovery run first.

### Nice-to-have UI cleanup

- Rename “Ask Truthwatcher” to “Questions” or “Read-only Answers” to avoid AI/LLM implications.
- Add copy buttons for IDs and API examples.
- Add a route-level “what this page proves” note.
- Add fixture/demo reset instructions.

## 9. Release-Blocking Bugs Or Inconsistencies

1. `docs/poc-walkthrough.md` references `GET /api/v1/relationships`, but no such route is registered.
2. `internal/app.go` help text references `fixture://iosxr-pe`, while current IOS-XR fixtures use `fixture://iosxr-asr`.
3. `docs/concepts/discover-how-to-discover.md` says parser persistence is future/separate work even though it is implemented.
4. Discovery run detail UI fetches evidence but does not render evidence records.
5. Identity review has API support but no embedded UI path.
6. Audit persistence exists for discovery, but there is no audit read API or UI inspection path.
7. README and docs mix proof-of-concept, alpha, and broader platform language; v0.1.0 needs a tighter release definition.
8. `runParseDiscoveryRun` contains a duplicate unreachable database URL error return.
9. No release acceptance command/script validates the full fixture-backed evidence-to-graph workflow.
10. The deterministic “Ask” surface may be mistaken for an LLM/chat feature unless renamed or explained more strongly.

## 10. Smallest Ordered Implementation Plan To Reach v0.1.0

### Blocker plan

1. Fix obvious inconsistencies: remove the duplicate parse error return, correct the IOS-XR fixture name in help, and remove or implement the undocumented global relationships endpoint.
2. Add or update docs: v0.1.0 quickstart, expected output appendix, known limitations, identity review guide, audit guide, UI route guide, and prompt status page.
3. Add an acceptance script or Makefile target that runs the local fixture-backed flow against a configured database: migrate, fake discover, parse, API smoke checks, graph smoke check.
4. Add minimal audit read API: list audit records with filters for discovery run, evidence, request ID, action, and status.
5. Add audit UI page using the minimal audit API.
6. Expand discovery run detail UI to show evidence rows and raw output drawer links.
7. Add identity review UI page for pending queue, candidate details, evidence links, and non-destructive review actions.
8. Tighten graph and empty states so a new user knows when parsing is required.
9. Rename or clarify “Ask Truthwatcher” as deterministic read-only answers with no external LLM calls.
10. Run and record the v0.1.0 acceptance workflow, then tag only if the README path matches the passing workflow.

### Nice-to-have after blockers

1. Add screenshots to docs.
2. Improve graph layout and relationship labels.
3. Add fixture reset/seed UI actions if they remain local and safe.
4. Add parser/fixture authoring docs.
5. Archive completed prompts into a historical folder.

## Release Readiness Verdict

Truthwatcher is **not yet v0.1.0 usable** as-is because the evidence-first kernel exists but the user-facing release path is incomplete. It can become v0.1.0 usable with a small stabilization pass focused on documentation accuracy, demo reproducibility, evidence/audit/identity UI inspection, and removal of mismatches that would break a new user's trust.

The recommended v0.1.0 label is:

> **Truthwatcher v0.1.0: local fixture-backed evidence-to-graph proof of concept.**

That label is honest, valuable, and consistent with the repository's strongest current capability.
